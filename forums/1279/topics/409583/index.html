    Hello all

    I would like to introduce SingaporeBus (https://projects.forum.nokia.com/SingaporeBus):
    An application to help find nearby bus stops, bus arrival times and bus itineraries in Singapore, based on information extracted from publictransport@sg, Iris next bus and TransitLink online services. Includes map and paths of the bus services. Developed for N95 but the app should be compatible with most touch and non-touch Symbian S60 phones (Nokia, SonyEricsson, Samsung...).

    So far the application has only been tested with N95 and on the s60 emulators (3rd edition FP1/2 and 5th edition). Please share your successfull and unsuccessfull experiences here or on the project homepage!

    SingaporeBus is totally free (except carrier charges if you use packet data of course...) and is open source. If anyone's interested I would more than welcome volunteers to help grow and maintain the application as I will soon have less free time.

    Cheers!

    sergiogiogio

    Hello n4l * I agree, I am also used to red call key placing apps into background instead of closing them. I initially did not pay attention to this behavior because I did not code it. After some research it seems to now be the new expected behavior(http://wiki.forum.nokia.com/index.php/KIS000294_-_Call_termination_key_behavior_has_changed) and the IDE (Carbide) automatically generates the code to comply to that behavior. To place the application into the background there's now always the menu key :) * The search function should search substrings, could you check again? One thing to keep in mind though when searching is that the database is full of abbreviations such as "rd" for road, "jln" for jalan, etc. With a device with a keyboard, such as yours I believe, you can just start typing and the search field will show immediately (there's no need to go into Options > Search, this menu option was added for touch devices). I have added to the application homepage the list of all the shortcuts in each screen. * Regarding showing less bus stops, I can understand it could make the application easier to use in certain cases. But I fear it can also make the application less intuitive or quick to use in other cases. Furthermore, as you guessed it would be a big development due to the way the app is currently written :) The app is written in Symbian C++ with native avkon controls which are extremely painful to use in my experience. That's one of the reason there are only very few screens in SingaporeBus. With the many ideas you have, you would be much welcome to participate in the development of the app!! * Regarding the jumping screen, I definitely want to resolve this issue but I am still thinking of the best solution. GPS is the most accurate but it sometimes takes long to complete (more than 10s), it does not always work (indoors), and some devices do not have it. Even if the app could detect whether a GPS is available, with a 10s timeout I don't like that you need to wait that long in order to use the application (maybe you do not care about your current location, you actually want to do a search). And after 10s, what should the app do, query for google and network and cancel the GPS to avoid confusion due to a later GPS position? SingaporeBus v1.2.0 has just been released with some improvements over the jumping screen issue. The basic idea is that the app will maintain the selected bus after a new position is resolved. this should help with the screen jumping but is not a definitive solution. Cheers! sergiogiogiogio
